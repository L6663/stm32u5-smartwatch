# STM32U5 智能手表项目审查与第 3 轮迭代报告

**版本定位：** v0.3.0-alpha-review  
**审查对象：** projectlvgl  
**审查日期：** 2026-07-11  
**目标：** 在不推翻现有架构的前提下，确认真实完成度、识别稳定性风险，并完成可验证的最小安全迭代。

---

## 1. 结论

该工程不是空壳，也不是完整智能手表成品，而是一个已经进入 **Alpha 原型阶段** 的 STM32U5 智能穿戴平台。

工程已经建立以下关键闭环：

- STM32U575 外设初始化与启动诊断；
- ILI9341 + SPI1 + GPDMA 的 LVGL 显示链路；
- FT6336 触摸和五页 UI；
- FreeRTOS 任务、队列及 LVGL 单线程调用约束；
- RTC、SHT20 和 MAX30102 的应用层数据通道；
- 健康页、运动页和天气页的界面更新机制。

但项目仍有明显的演示属性：步数、电量、天气、运动数据和来电业务尚未由真实硬件或网络驱动；血氧算法未标定；ESP8266、AP3216C 等代码资产没有纳入当前产品闭环。因此当前版本适合用于求职项目演示、工程架构讲解和后续迭代，不宜表述为“功能全部完成”或“量产级产品”。

第 3 轮已修复一项高风险并发问题和三项时序/输入问题，未对 UI 架构和现有功能做破坏性重构。

### 2.2 RTOS 资源

| 任务 | 优先级 | 栈空间 |
|---|---:|---:|
| LVGLTask | High | 8192 B |
| HealthTask | AboveNormal | 8192 B |
| SensorTask | Normal | 4096 B |
| SystemTask | BelowNormal | 3072 B |

- LVGL Heap：96 KB。
- RTOS Tick：1000 Hz。
- 显示缓冲：两个 320×20 RGB565 绘图缓冲，加一个同尺寸 DMA 缓冲，合计约 38.4 KB。

### 2.3 历史构建资源占用

上传工程中最近的 Keil Map 显示：

- RO：1,267,904 B（约 1238.19 KB）
- RW + ZI：211,712 B（约 206.75 KB）
- ROM：1,268,188 B（约 1238.46 KB）

该结果说明原工程在上传前曾生成完整固件，但不等同于本次 V3 修改已通过 Keil 链接。

## 3. 功能真实性审查

### 3.1 已形成真实链路

- LCD 初始化、LVGL 绘制、DMA 完成回调和刷新完成通知。
- FT6336 触摸输入与页面滑动。
- 五页 SquareLine 界面及统一页面导航。
- RTC 时间读取、数字时间和表盘更新。
- SHT20 温湿度读取。
- MAX30102 初始化、FIFO 采样、波形、BPM 计算和 UI 更新框架。
- 非 LVGL 任务通过消息队列更新界面，未直接跨任务调用 LVGL。

### 3.2 部分真实、部分回退

- SHT20 读取失败后会自动使用缓慢变化的演示温湿度；目前用户界面不会明显提示“模拟数据”。
- MAX30102 的 BPM 逻辑具备真实采样路径，但仍需在具体传感器、佩戴结构和人手环境中调参。
- 血氧使用经验近似公式，源代码已明确标注需要与参考血氧仪做光学/结构标定。
- RTC 使用 LSI，能运行但精度不适合作为长期准确时钟；缺少网络校时。

### 3.3 纯演示或未接入

- 步数由伪随机数递增。
- 电池电量固定为 `86%`。
- 天气由本地表每 30 秒轮换。
- 运动距离、热量和运动心率来自演示模型。
- 来电按钮没有接听/拒接业务回调。
- ESP8266 源码存在于独立目录，但没有形成当前主工程的联网链路。
- AP3216C 驱动存在，但应用层没有使用。

## 4. 风险分级

### P0：I²C1 并发访问

**原问题：** SHT20 和 MAX30102 使用 `i2c1Mutex`，FT6336 触摸却直接调用 HAL I²C。健康采集、环境采集和触摸轮询可能同时访问 I²C1，导致 `HAL_BUSY`、数据帧混合、触摸偶发失效或任务等待异常。

**V3 处理：** FT6336 已纳入同一个 `i2c1Mutex`，并加入互斥锁超时计数与 I²C 错误计数。

**剩余风险：** AP3216C 若后续启用，也必须迁移到同一总线访问层，不能继续直接读写 I²C1。

### P1：触摸坐标边界

**原问题：** 横屏坐标使用 `x = 320 - raw_y`，当 `raw_y=0` 时会得到 320，超出 0–319 的有效范围。

**V3 处理：** 修改为 `x = 319 - raw_y`，并对原始 X/Y 做范围限幅。

### P1：触摸采样不是原子帧

**原问题：** 一个触点由多次单字节 I²C 读取拼装，读取期间控制器可能更新触点寄存器，造成 ID、X、Y 来自不同采样时刻。

**V3 处理：** 从 `TD_STATUS` 开始一次连续读取 7 字节，统一解析触点状态、ID、X、Y。

### P1：系统周期漂移

**原问题：**

- LVGL 每轮固定执行 `lv_tick_inc(5)`，任务受阻时 LVGL 时间会慢于真实时间。
- SensorTask 在执行 SHT20 转换后再 `osDelay(1000)`，每个周期会额外叠加传感器转换时间。
- ClockApp 使用相对延时，长期运行会累积调度误差。

**V3 处理：**

- LVGL Tick 根据 RTOS 实际 Tick 差值推进。
- SensorTask 和 ClockApp 使用 `osDelayUntil` 的绝对周期。

### P1：功能状态可能被误解

**问题：** 页面均有数据显示，但其中多项为模拟值；若 README 只写“实现智能手表所有功能”，会在项目展示或面试追问时产生可信度风险。

**V3 处理：** 新增 README 和功能完成度表，对真实、部分实现、模拟、未接入进行明确区分。

### P2：固件和仓库体积偏大

三个 320×240 背景图 C 数组各约 1.16 MB 源码，链接后的像素数据约 230 KB/张，是 ROM 占用的重要来源。LVGL 自带 demos 目录约 57 MB，未被当前 Keil 工程引用，却显著放大仓库和压缩包。

**V3 处理：** 清理版源码包移除未引用的 `LVGL/demos` 与本地构建产物；未直接修改图片格式，以避免改变视觉效果。

**后续建议：** 将全屏 RGB888 图片转换为 RGB565、索引色或 LVGL 压缩资源，预计三张全屏图可节省约 230 KB ROM，具体值以重新链接结果为准。

### P2：遗留模块与重复实现

- Keil 工程仍编译旧的 `algorithm.c`、`max30102.c`、`max30102_app.c`，但当前实际业务使用 `health_app.c` 内的新实现，链接器会移除大量未使用代码。
- `cJSON.c`、AP3216C 驱动被纳入工程，但当前业务未使用。
- ESP8266 目录中的代码未加入主工程。

这不会必然造成运行错误，但增加维护成本和理解难度。应在下一轮明确“保留待接入”或“移出主构建”的边界。

### P2：背光 PWM 待核实

TIM2 CH3 已配置 PWM，但全工程未检索到 `HAL_TIM_PWM_Start`。若 PB10 确实用于背光 PWM，则复位后背光状态可能依赖外部电路或默认 GPIO 状态。由于缺少当前板级原理图确认，本轮不盲目启动 PWM，避免极性错误或与实际引脚用途冲突。

## 6. 验证结果与边界

### 已完成

- 对关键任务、LVGL 端口、触摸、显示、SHT20、MAX30102、RTC、运动和 UI 文件进行静态审查。
- 检查 Keil 工程文件中的模块引用关系。
- 检查历史 Map 的 ROM/RAM 占用。
- 使用 Clang C11 对修改后的关键 C 文件执行语法检查，未发现语法错误。
- 清理包中移除了未被 Keil 工程引用的 LVGL demos 与陈旧构建输出。

### 当前无法证明

- 本环境没有 Keil ARM Compiler 6，未执行目标架构完整编译、链接和烧录。
- 未连接真实 STM32U575 板卡，无法确认 I²C 波形、传感器响应、DMA 中断、触摸方向和背光电气行为。
- 历史 `.axf/.hex/.map` 属于 V3 修改之前，不能作为 V3 固件验证结果。

因此，V3 的结论是：**代码审查与主机语法检查通过，目标板回归仍是发布前必做项。**

## 7. 目标板回归清单

1. 全量 Rebuild，确认 0 Error；对新增 Warning 单独记录。
2. 冷启动、软件复位、断电重启各 10 次，确认 LCD 均能亮起。
3. 连续左右滑动五页 5 分钟，确认无卡死、错页和坐标跳变。
4. 点击屏幕四角和边缘，确认坐标不越界。
5. 同时运行 MAX30102、SHT20，并连续触摸 10 分钟：
   - `g_ft6336_i2c_error_count` 不持续增长；
   - `g_ft6336_mutex_timeout_count` 不持续增长；
   - SHT20/MAX30102 自身错误计数无异常上升。
6. 观察 `g_lcd_dma_start_error_count`、`g_lcd_dma_transfer_error_count` 和 `g_lcd_dma_busy_error_count`。
7. 断开 SHT20，确认系统不阻塞，同时明确验证 UI 是否进入模拟值。
8. MAX30102 无手指、有手指、移动手指分别测试，记录 BPM 稳定时间、误触发和血氧偏差。
9. 重启后检查 RTC 时间；连续运行 24 小时评估 LSI 漂移。
10. 核对 PB10/背光原理图，再决定是否启动 TIM2 PWM。
